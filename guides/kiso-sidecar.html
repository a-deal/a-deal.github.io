Sidecar Architecture: Zero-Downtime Deploys

How we split the Kasane v1 API from the coaching gateway so we can iterate on the shared data layer without taking Milo offline.

01
The Problem

The health-engine gateway runs as a single Docker container on Mac Mini. It serves Milo's MCP tools, wearable auth, and the new Kasane v1 API. Any code change requires a Docker rebuild, which takes Milo offline. Real users (Paul, Dad, Mike, Patrick) depend on Milo being available.

The v1 API is the most actively developed surface right now. We need to iterate fast without risking Milo's uptime.

02
The Architecture
Docker container (port 18800)         Python process (port 18801)
├── MCP tools (/api/{tool})           ├── v1 API (/api/v1/*)
├── Auth routes (/auth/*)             └── Kasane sync endpoint
├── Garmin/Oura/WHOOP integrations
└── OpenClaw gateway connection
         │                                      │
         └───── shared data directory ──────────┘
                /Users/andrew/src/health-engine/data/
                    ├── kasane.db (SQLite)
                    ├── users/ (CSVs)
                    └── admin/ (audit logs)

Two independent processes, one shared data directory. Either can restart without affecting the other. The Docker container handles the coaching agent. The sidecar handles the iOS sync API.

03
What Lives Where
ServicePortProcessClients
Coaching gateway18800Docker (auto-restart)Milo (OpenClaw), iOS Shortcuts, wearable auth
Kasane v1 API18801LaunchAgent (auto-restart)Kasane iOS app (SyncService)

Both read/write the same SQLite database and CSV files. No replication, no sync between them. SQLite handles concurrent access via WAL mode. The coaching gateway writes health data (meals, weight, labs, wearables). The v1 API writes habits, check-ins, and focus plans. Both can read everything.

04
Operations

Restart the v1 API (no Milo impact):

ssh mac-mini 'launchctl unload ~/Library/LaunchAgents/com.kiso.v1api.plist'
ssh mac-mini 'launchctl load ~/Library/LaunchAgents/com.kiso.v1api.plist'

Deploy code changes to v1 API:

ssh mac-mini 'cd ~/src/health-engine && git pull origin master'
ssh mac-mini 'launchctl unload ~/Library/LaunchAgents/com.kiso.v1api.plist'
ssh mac-mini 'launchctl load ~/Library/LaunchAgents/com.kiso.v1api.plist'

Check status:

curl http://mac-mini:18801/health        # v1 API
curl http://mac-mini:18800/health        # coaching gateway

View logs:

ssh mac-mini 'tail -f /tmp/kiso-v1.log'

06
Cloud Migration Path

When we outgrow the Mac Mini (more than 10 active users, need uptime guarantees, or Paul needs to deploy without Andrew):

Phase 1 (now): Mac Mini sidecar. Free, fast, good enough for concierge beta.

Phase 2: Deploy the v1 API to Fly.io or Railway. The sidecar script becomes the deployment artifact. The Docker gateway stays on Mac Mini for Milo. Kasane iOS points to the cloud URL.

Phase 3: Move everything to cloud. Docker gateway migrates to managed hosting. Mac Mini becomes a development/testing machine only.

Each phase is a configuration change, not a rewrite. The API contract stays the same.

Two processes, one data directory. Deploy the API without touching the coach. Iterate fast, break nothing.
